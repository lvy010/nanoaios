这份文档用于对齐 **Memory System 的背景、设计原则与架构要点**。

目标不是把所有实现细节一次性写完，而是先明确哪些是长期稳定的共识，哪些是后续展开细化的方向。


1. ### 背景：为什么需要工程化 Memory
    

- 真实场景下，Agent 的“记忆”不是无限长上下文，而是**在有限 context window 与成本约束下**，把外部信息以可控方式注入到当前推理中。
    
- 因此工程化 Memory 的核心问题不是“存储”，而是：
    
    - **注入什么**
        
    - **何时注入**
        
    - **以什么粒度注入**
        
    - **如何保证可控**
        
- 更具体地说，Memory System 的目标是：在有限预算下，稳定地完成 **选择、排序、压缩、引用** 这四件事，从而形成当前可推理的 context。
    

2. ### 第一性原理：主动记忆是“概率正确”，异步进程是“确定性地基”
    

#### 2.1 Agent 主动记忆的边界

- Agent 主动“记”的方式类似人，只能做到：**以更高概率记录对未来有用的信息**。
    
- 由于未来任务未知，它无法保证“必然记对”。
    
- 现阶段模型本身也还不擅长稳定地“记得好”，因此系统必须允许记忆犯错，而不是把一次错误抽象永久固化。
    
- 这意味着主动记忆应被设计成：
    
    - 可回滚
        
    - 可纠错
        
    - 可再索引
        
    - 可与原始记录对照
        

#### 2.2 异步进程的价值

异步进程至少承担两件确定性工作：

1. **沉淀原始信息**
    
    1. 记录工作中的原始信息，类似人的笔记、法庭的原始记录。
        
    2. 在短期内尚未获得理想化 AgentFS 的情况下，文件系统化组织是最现实、最稳妥的基础设施。
        
2. **构建可用索引**
    
    1. 为文件系统提供好的索引，引导 Agent 使用。
        
    2. 不管是 RAG、结构化索引还是 tag，本质上都在做一件事：**提供索引，使 agent 能更快、更准地找到可能有用的信息**。
        

3. ### 设计原则
    

#### P1. 证据优先

- 任何高层结论，例如 summary、lesson、root memory，都应尽量能指回原始记录，至少保留明确的来源与指针。
    
- 没有证据的高层记忆很难 debug，也很难判断是“记错了”还是“用错了”。
    

#### P2. 分层与预算

- context window 有限，因此 Memory 必须分层。
    
- 一个实用的基本分层是：
    
    - **核心 memory**
        
    - **近期 memory**
        
    - **相关 memory**
        
- 每层都应有自己的注入预算、更新频率与重要度，而不是平铺地 top-k 塞进上下文。
    

#### P3. 冲突可见

- Memory 作为信息源天然会冲突。
    
- 冲突可能来自时间、来源或抽象层次的不一致。
    
- 系统不应只靠“最新优先”粗暴覆盖，而应让冲突对系统内部可见，便于后续纠错与治理。
    

#### P4. 渐进式披露

- Memory 不可能一次性完整注入，也不可能预先知道未来每一步会需要什么。
    
- 因此系统必须支持渐进式披露：先注入最必要的部分，再在需要时按需加载原始记忆或更细粒度的证据。
    

#### P5. 成本内生

- 速度与成本是设计目标的一部分，而不是事后优化项。
    
- 一个 memory 系统的价值，不只是“能不能记住”，还包括：
    
    - 是否减少重复探索
        
    - 是否降低 token 成本
        
    - 是否减少无效检索
        
    - 是否提升响应速度
        

4. ### 架构要点：工程化 Memory = 外部存储 → 受控注入 → 当前 Context
    

#### 4.1 外部存储不是单一层，而是分层结构

- **原始记录层**：events、traj、tool result、关键文件片段、操作日志
    
- **索引层**：embedding、keyword、结构化索引、tag
    
- **注入层**：规则注入、检索注入、按需二次加载
    

#### 4.2 注入过程是系统的核心能力

Memory System 的关键不是“有没有存下来”，而是能否稳定地完成：

1. 选择
    
2. 排序
    
3. 压缩或提炼
    
4. 引用与追溯
    

#### 4.3 渐进式注入的基本形态

- **规则注入**：注入核心 memory、近期 memory、全局约束
    
- **按需检索与二次加载**：基于 query 、索引与原始记录，多路获取信息，agentic 地获取信息；推理过程中发现信息缺口时，再回到原始记录层补充
    

5. ### Code Agent 的先验：两类 Memory，分别治理
    

#### 5.1 Code memory

更接近事实与结构，例如：

- 技术栈
    
- 构建与测试入口
    
- 模块边界
    
- 关键约束与契约
    
- 常见定位入口
    

#### 5.2 Lessons

更接近交互中沉淀的指导性经验，例如：

- 做任务时的流程经验
    
- 某类问题的检查清单
    
- 过去踩坑形成的总结
    

这类信息更抽象、更容易迁移，但也更容易出错，因此更需要：

- 明确适用范围
    
- 关联来源或证据
    
- 保守更新
