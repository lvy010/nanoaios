# Extension First：防止 Agent 核心循环膨胀的架构原则

> **核心命题**：任何新功能，第一反应必须是——能用 Extension 实现吗？

---

## 一、为什么 Agent 核心循环容易膨胀

Agent 框架的核心循环天然承担三项职责：

1. **调度**：管理 LLM 调用、工具执行、中断处理的流程
2. **状态**：维护会话、消息、重试等运行时状态
3. **协议**：与 LLM Provider、工具执行器、上下文管道等模块交互

当开发者需要添加新功能时，最直观的做法是 **直接在核心循环里加代码**：

- 添加调试日志：`if (enableDebug) { log(...) }`
- 添加权限控制：`if (requireConfirmation) { askUser(...) }`
- 添加技能系统：`if (enableSkills) { injectSkillPrompts(...) }`

每次这样做，核心循环就增长几十行。久而久之：

- **可读性崩溃**：单文件数千行，夹杂调度逻辑和业务特性
- **测试困难**：无法独立测试新功能，必须启动完整的核心循环
- **耦合加深**：新功能依赖核心循环内部状态，反之亦然
- **维护成本激增**：修改一个功能可能影响循环逻辑，改循环可能破坏特性

---

## 二、Extension First 原则

### 2.1 核心思想

**将核心循环视为不可变的调度内核，所有功能扩展通过 Extension 注入。**

Extension 不是核心循环的替代品，而是它的 **扩展点使用者**：

- 核心循环提供扩展点（Hook Point），Extension 在这些点注册处理器
- Extension 与核心循环解耦，可独立开发、测试、组合
- 核心循环只保留调度逻辑，不包含任何业务特性

### 2.2 设计机制

```
┌───────────────────────────────────────────────┐
│            核心循环 (Agent Loop)              │
│  - LLM 调用调度                               │
│  - 工具执行协调                               │
│  - 中断处理                                   │
│  - 流式响应管理                               │
└───────────────┬───────────────────────────────┘
                │
                ▼
┌───────────────────────────────────────────────┐
│           扩展点引擎 (Hook Engine)            │
│  - LLM 调用前                                 │
│  - LLM 响应后                                 │
│  - 工具执行前                                 │
│  - 工具执行后                                 │
└───────────────┬───────────────────────────────┘
                │
        ┌───────┴───────┬───────────┐
        ▼               ▼           ▼
  ┌──────────┐   ┌──────────┐  ┌──────────┐
  │Extension │   │Extension │  │Extension │
  │  调试    │   │  技能    │  │  权限    │
  └──────────┘   └──────────┘  └──────────┘
```

**关键机制**：

1. **Hook Point（扩展点）**：核心循环在关键节点触发已注册的处理器
2. **Extension 接口**：统一的生命周期（设置 → 激活 → 停用）和状态隔离
3. **依赖注入**：Extension 通过管理器注册，运行时由框架自动激活

---

## 三、达成的效果

- **核心循环**：循环调度 + 流控制
- **Extension**：业务特性 + 策略决策
- **抽象接口**：基础设施抽象

### 3.1 与传统做法的对比

| 维度 | 传统做法（特性内置） | Extension First |
|------|---------------------|----------------|
| **核心循环代码量** | 随特性线性增长 | 保持稳定 |
| **新功能开发** | 需要理解整个循环 | 只需理解 Hook 接口 |
| **测试独立性** | 必须启动完整循环 | Extension 可独立测试 |
| **并行开发** | 高冲突风险 | 多人开发不同 Extension，零冲突 |
| **组合灵活性** | 特性开关互相干扰 | Extension 正交组合 |
| **回归风险** | 修改核心影响所有特性 | Extension 失败不影响循环 |

### 3.2 实际案例

典型 Agent 框架已有的 Extension（无需修改核心循环）：

| 需求 | Extension 实现方式 |
|------|--------------------|
| 在系统提示中注入内容 | LLM 调用前 Hook → 修改 prompt |
| 过滤/修改消息历史 | LLM 调用前 Hook → 修改 messages |
| 拦截工具调用（权限控制） | 工具执行前 Hook → 返回拒绝或请求用户确认 |
| 转换工具调用结果 | 工具执行后 Hook → 修改 result |
| 后处理 LLM 输出 | LLM 响应后 Hook → 修改输出内容 |
| 添加新工具 | Extension 设置阶段注册工具 |
| 维护会话级状态 | Extension 激活阶段 + 状态 API |
| 监听运行时事件 | Extension 激活阶段订阅事件总线 |


## 四、何时不该用 Extension

以下情形**不**适合 Extension，应选择其他路径：

| 情形 | 正确路径 |
|------|----------|
| 修改 LLM 调用的基础逻辑（流解析、重试） | 修改 LLM Provider 抽象实现 |
| 改变消息历史的构建方式 | 修改上下文管道（Context Pipeline） |
| 新增事件总线的事件类型 | 修改事件系统的类型定义 |
| 需要新的数据持久化方式 | 添加新的存储适配器实现 |
| 工具不属于 Agent 能力，而是独立命令 | 直接在应用层处理 |

---

## 五、总结

**Extension First 原则的本质是：通过架构约束防止核心代码膨胀。**

它不是技术创新，而是工程纪律——通过明确的边界、清晰的扩展点、文档化的约束，让团队在 "方便" 和 "正确" 之间始终选择后者。

**核心洞察**：

1. **膨胀不可避免**：没有约束，任何核心代码都会随着需求膨胀
2. **扩展点是解药**：提供扩展机制，让新功能在外围生长
3. **文档是保障**：通过明确的原则和流程，而非技术限制，维护架构纪律

**最终目标**：核心循环成为稳定的、可预测的调度内核，所有创新在 Extension 层发生。
